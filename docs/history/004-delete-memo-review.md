# History

## 改善点

Codexによる実装を行う際のプロンプトにおいて、コードを変更する前に「実装アプローチ」の提示を必須とする。

## 理由

実装アプローチの提示は、設計プロセスを置き換えることを目的としない。

設計に関する決定は、Issue作成および設計レビューの段階で、HumanとChatGPTが行う。

実装アプローチは、コードを変更する前に、承認済みの設計をCodexが正しく理解しているか確認するために使用する。

## 対応策

実装用プロンプトに以下の要件を追加する。

> 実装を行う前に、実装アプローチ、変更予定のファイル、および前提条件を説明してください。  
> もしアプローチが安全でない、またはIssueの内容と整合しないと思われる場合は、実装を行わずに作業を中断し、その理由を説明してください。

## 改善点

開発フローに「設計」を明確な工程として追加する。

## 背景

これまでは以下の流れで開発を進めていた。

```text
Issue
↓
Prompt
↓
Codex
```

しかし、この流れでは設計と実装指示が混在しやすく、役割が曖昧になっていた。

## 決定

開発フローを以下のように変更する。

```text
Issue
↓
Design（Human + ChatGPT）
↓
Prompt（ChatGPT）
↓
Codex
```

設計はHumanとChatGPTが担当する。

設計レビューを完了してから、Codex用の実装プロンプトを作成する。

Codexは設計を決定する役割ではなく、承認済みの設計に基づいて実装する役割とする。

## 期待される効果

- 設計と実装の責務を明確に分離できる
- Codexへの実装指示が明確になる
- 手戻りを減らせる
- Trace Appでも同じ開発フローを適用できる

## 改善点

Codexへ渡す実装プロンプトに「Approved Design」セクションを追加する。

## 背景

これまではGitHub Issueの内容のみを基にCodexへ実装を依頼していた。

GitHub Issueには「何を実装するか」は記載されているが、「どのように実装するか」という設計方針は含まれていない。

そのため、Codexが設計と異なる実装方針を採用したり、不要なリファクタリングやコンポーネント分割を行う可能性があった。

## 決定

実装プロンプトに「Approved Design」セクションを追加する。

Approved Designには、HumanとChatGPTによる設計レビューで承認された実装方針を記載する。

CodexはApproved Designを実装の基準とし、設計変更は行わない。

## 期待される効果

- Codexによる不要な設計変更を防止できる
- 実装の一貫性を維持できる
- レビューコストを削減できる

## 改善点

Codexが実装内容を検証するための非破壊コマンドは、追加の確認なしで実行できるものとする。

## 理由

以下のコマンドはリポジトリやソースコードを変更せず、動作確認や情報取得のみを目的としている。

毎回実行許可を求めることは開発効率を低下させるため、Codexが必要に応じて実行できるものとする。

## 対応策

以下のようなリポジトリを変更しないコマンドは、Codexが必要に応じて実行してよい。

- npm run dev
- npm run build
- npm run lint
- curl
- cat
- ls
- pwd
- find
- grep

リポジトリや開発環境へ変更を加えるコマンドは、従来どおり明示的な指示がある場合のみ実行する。

例

- git commit
- git push
- git merge
- git rebase
- npm install
- npm update

## 期待される効果

- 不要な確認を減らせる
- 実装と検証をスムーズに進められる
- HumanとCodexのやり取りを削減できる
