# Approved Design — Issue #60

## Goal

Implement a shell-based workflow that creates a GitHub Pull Request from an AI-generated Pull Request Draft without requiring manual copy-and-paste.

The created Pull Request must contain:

- The title generated by the Pull Request Draft prompt.
- The generated Pull Request body.
- A base branch that defaults to `dev` and can be overridden.
- A head branch that defaults to the current working branch and can be overridden.

The command must preserve the Human review step required by Issue #60 before the Pull Request is created.

---

## Scope

The implementation will:

- Add a script for creating a Pull Request from a generated Markdown draft.
- Accept an Issue number or an explicit draft file path.
- Read the Pull Request title and body from the draft.
- Use `dev` as the default base branch.
- Allow the base branch to be overridden with a command option.
- Use the current Git branch as the default head branch.
- Allow the head branch to be overridden with a command option.
- Display the resolved title, draft file, base branch, and head branch before creation.
- Request Human confirmation before running the GitHub operation.
- Create the Pull Request with `gh pr create`.
- Display the created Pull Request URL.
- Reject invalid repository states and duplicate Pull Requests with clear errors.
- Add or update relevant tests and workflow documentation where required.

Proposed command interface:

```bash
./scripts/create-pull-request.sh 60
```

Override the base branch:

```bash
./scripts/create-pull-request.sh 60 --base main
```

Override the head branch:

```bash
./scripts/create-pull-request.sh 60 --head feature/example
```

Override both:

```bash
./scripts/create-pull-request.sh 60 --base release --head feature/example
```

Use an explicit draft file when necessary:

```bash
./scripts/create-pull-request.sh --draft docs/history/pr-draft-60.md
```

---

## Out of Scope

- Generating the implementation summary or Pull Request Draft inside the script.
- Creating or modifying GitHub Issues.
- Creating, renaming, or switching Git branches.
- Committing or pushing Git changes.
- Automatically merging the Pull Request.
- Assigning reviewers, assignees, labels, projects, or milestones.
- Running Repository Review.
- Updating `.ai/context.md` or `.ai/state.json`.
- Modifying application code.
- Creating Pull Request review comments after the Pull Request is opened.
- Supporting Git hosting providers other than GitHub.

## Design Policy

### 1. Keep draft generation and Pull Request creation separate

The existing Pull Request Draft prompt remains responsible for generating the title and body.

The new script is responsible only for validating the generated draft and creating the Pull Request from it.

This preserves the existing separation of responsibilities:

```text
AI generates Pull Request Draft
↓
Human reviews the draft
↓
Script resolves Pull Request metadata
↓
Human confirms creation
↓
GitHub Pull Request is created
```

### 2. Define a stable draft-file contract

The script must not attempt to infer a title from arbitrary Markdown.

The Pull Request Draft template should produce a predictable structure containing one explicit title field followed by the Pull Request body.

Preferred structure:

```md
# Pull Request Draft

## Title

docs: update repository workflow documentation

## Body

## Summary

...

## Changes

...

## Verification

...
```

The script will:

- Read the first non-empty line after `## Title` as the Pull Request title.
- Read all content after `## Body` as the Pull Request body.
- Reject the draft when either section is missing or empty.

This format avoids fragile parsing based on the first Markdown heading and prevents the title from being included in the Pull Request body.

### 3. Use explicit branch defaults with optional overrides

Branch resolution order:

**Base branch**

1. `--base <branch>` when provided.
2. `dev` otherwise.

**Head branch**

1. `--head <branch>` when provided.
2. Current Git branch otherwise.

The resolved branches must always be displayed before confirmation.

### 4. Validate before creating the Pull Request

The script should fail before calling `gh pr create` when:

- Required arguments are invalid.
- The draft file does not exist.
- The draft format is invalid.
- `git` or `gh` is unavailable.
- GitHub CLI authentication is unavailable.
- The current directory is not a Git repository.
- The resolved head branch is `main` or `dev`, unless explicitly allowed by repository policy.
- The base and head branches are identical.
- The head branch has not been pushed or cannot be resolved on the remote.
- A Pull Request already exists for the resolved head branch.

### 5. Preserve Human approval

The script should display a concise creation preview and require an explicit confirmation such as:

```text
Create this Pull Request? [y/N]
```

Any response other than an affirmative answer must cancel without creating a Pull Request.

A non-interactive bypass such as `--yes` is not included in this Issue because the approved workflow requires Human confirmation.

### 6. Avoid unnecessary temporary files

The implementation should create a temporary body file only when required by `gh pr create --body-file`.

Any temporary file must be removed with a cleanup trap on success, cancellation, or error.

### 7. Follow existing script conventions

The implementation should reuse existing helpers under `scripts/lib/` when they already cover:

- Error output
- Command validation
- Repository validation
- Branch resolution
- Issue-number validation

New shared helpers should be introduced only when they reduce meaningful duplication.

## Architecture Impact

### Application Architecture

No impact.

### Repository Workflow

The Pull Request creation step changes from manual copy-and-paste to a repository script.

Updated workflow:

```text
Implementation completed
↓
Verification completed
↓
Pull Request Draft generated
↓
Human reviews the draft
↓
create-pull-request.sh
↓
Human confirms resolved metadata
↓
Pull Request created
```

### AI Workflow

The AI-generated Pull Request Draft becomes a machine-readable input to the repository workflow.

A stable contract between the Pull Request Draft prompt and the creation script is required.

### Documentation

The Pull Request Draft format and script usage may require updates in:

- The Pull Request Draft prompt template.
- The Playbook or relevant workflow documentation.
- Script usage documentation, if such documentation already exists.

### AI Context

No direct impact.

## Implementation Plan

1. Inspect the existing shell scripts and helpers under `scripts/`.
2. Confirm the current Pull Request Draft storage location and naming convention.
3. Update the Pull Request Draft prompt template to emit the agreed `Title` and `Body` sections if the current format is not already machine-readable.
4. Add `scripts/create-pull-request.sh`.
5. Implement argument parsing for:
   - Issue number
   - `--draft`
   - `--base`
   - `--head`
   - `--help`
6. Resolve the draft file using the Issue number or explicit path.
7. Parse and validate the title and body.
8. Resolve the base and head branches using the defined precedence.
9. Perform command, authentication, repository, remote-branch, and duplicate-PR checks.
10. Display the Pull Request creation preview.
11. Request explicit Human confirmation.
12. Create the Pull Request with:

   ```bash
   gh pr create \
     --base "$base_branch" \
     --head "$head_branch" \
     --title "$pr_title" \
     --body-file "$body_file"
   ```

13. Display the returned Pull Request URL.
14. Add tests covering parsing, defaults, overrides, cancellation, and validation failures.
15. Update relevant workflow documentation.
16. Set executable permission on the new shell script.
17. Run the planned verification.

## Risks

- A change to the Pull Request Draft format could break parsing.
- Draft file naming or storage conventions may not yet be standardized.
- A branch may exist locally but not on the configured remote.
- `gh pr create` may select an unintended repository when multiple remotes exist.
- Directly testing GitHub operations could create unwanted Pull Requests.

Mitigations:

- Use an explicit draft-file contract.
- Validate the selected repository and remote state before creation.
- Mock `gh` in automated tests.
- Require Human confirmation before the real GitHub operation.
- Detect existing Pull Requests before creation.
